Waypoint emits OpenTelemetry metrics over OTLP to any compatible collector. <TestRef name="TestConfig_Defaults" />

```toml
[metrics]
endpoint = "http://otel-collector:4318"
protocol = "http"          # "http" (default) or "grpc"
interval = "30s"
temporality = "delta"      # "delta" (default, recommended for Datadog) or "cumulative"

[metrics.enable]
"waypoint.conn.active"   = ["listener", "mode"]
"waypoint.conn.total"    = ["listener", "mode"]
"waypoint.auth.attempts" = ["listener"]
"waypoint.bytes.read"    = ["listener", "user"]
"waypoint.bytes.written" = ["listener", "user"]
```

## Per-metric opt-in

Metrics are emitted only when explicitly listed under `[metrics.enable]`. <TestRef name="TestConfig_IsMetricEnabled" /> <TestRef name="TestEnabledMetrics_Record" /> <TestRef name="TestNonEnabledMetrics_UseNoop" /> The key is the metric name; the value is a **tag allow-list**:

- `["listener", "mode"]` — only these tags are attached. <TestRef name="TestAttrs_FiltersTags" /> <TestRef name="TestConfig_TagsForMetric" />
- `["*"]` — all available tags. <TestRef name="TestAttrs_Wildcard" />
- `[]` — no tags (the metric is a single scalar). <TestRef name="TestAttrs_EmptyTagList" />

Unknown tag names in the allow-list are flagged at load time. <TestRef name="TestAttrs_FilterVerification" /> <TestRef name="TestAttrs_UnknownMetric" /> This makes cardinality predictable: a metric like `waypoint.bytes.read` tagged by `user` will produce one time series per user; if that's too many, narrow the tag list to `["listener"]`.

| `waypoint.auth.failures` | counter | `listener` |

| `waypoint.bytes.read` | counter | `listener`, `user` |
| `waypoint.bytes.written` | counter | `listener`, `user` |
| `waypoint.limit.violations` | counter | `listener`, `limit_type` |
| `waypoint.provision.total` | counter | `listener` |

| `waypoint.system.listeners` | gauge | — |

## Temporality

Default is `delta` because Datadog and most modern collectors prefer it for counters and histograms. <TestRef name="TestConfig_TemporalitySelector" /> Use `cumulative` if your backend (e.g. Prometheus) wants monotonic counters.

## Endpoint

`endpoint` is required to enable any metrics export — without it, nothing is sent regardless of `[metrics.enable]` contents.
